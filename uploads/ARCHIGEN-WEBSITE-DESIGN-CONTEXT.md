# ArchiGen AI Website and Design Context

## Purpose of This Document
This file is meant to guide the **building of the ArchiGen AI website**.
It focuses only on website direction, design intent, content structure, UX priorities, and what the site needs to communicate.

It intentionally avoids broader company org details, agent roles, and background that do not help build the site.

## What the Website Needs to Be
ArchiGen AI should not feel like a generic AI tool directory or a cold SaaS product.

The website should feel like:
- a premium editorial platform for architects and designers using AI,
- a curated design intelligence site,
- a visual-first publication with strong taste,
- and a trusted place to discover tools, workflows, and outputs that actually matter.

The website needs to communicate:
- curation,
- authority,
- architectural relevance,
- visual quality,
- practical usefulness,
- and modern taste.

## Core Positioning the Website Must Signal
The site should position ArchiGen AI as:
- **architecture-first**, not general AI,
- **editorial**, not a cluttered tool warehouse,
- **proof-heavy**, not hype-heavy,
- **image-led**, not mostly text-led,
- **curated**, not exhaustive,
- **warm and premium**, not cold and generic.

## What Quincy Wants the Site to Feel Like
Quincy has been clear that the site should feel:
- warm,
- editorial,
- luxurious in tone,
- clean,
- modern,
- image-led,
- and grounded in real architectural outputs.

He does **not** want it to feel:
- cold,
- sloppy,
- generic,
- too close to the old directory version,
- like a startup template,
- or like a big list of tools with prettier styling.

## The Core Website Job
The homepage should not try to do everything equally.
Its primary job is to establish ArchiGen AI as the trusted front door for AI in architectural visualization.

That means the site should:
1. make a strong first visual impression,
2. show taste and relevance immediately,
3. present AI as part of real architectural workflow,
4. guide people into content, tools, and proof,
5. and make the platform feel curated rather than dumped together.

Secondary jobs:
- support tool discovery,
- support article reading,
- support future newsletter/community growth,
- support long-term monetization.

## Audience the Website Must Speak To
The site is for:
- architects,
- architectural designers,
- concept designers,
- archviz artists,
- students,
- and design-forward firms exploring AI workflows.

This audience wants:
- speed,
- clarity,
- visual proof,
- tool relevance,
- workflow guidance,
- better outputs,
- and less noise.

They do not want generic AI marketing language.
They want to see:
- what works,
- what it looks like,
- which tools matter,
- and how to use them in real design workflows.

## What the Homepage Should Prioritize
The homepage should act more like a premium publication front page than a giant directory.

It should likely prioritize, in this order:
1. **Strong visual hero with real architectural imagery**
2. **Clear positioning statement**
3. **Editorial guidance on what to explore first**
4. **Featured outputs / featured workflows / featured tools**
5. **Selected high-value content blocks**
6. **Structured paths into the deeper library**

It should avoid leading with:
- giant filter systems,
- overwhelming tool grids,
- too many category boxes,
- abstract feature language,
- or proof-free claims.

## The Right Website Identity
The right identity is likely:
- **editorial front door** on the homepage,
- **structured tool and article library** underneath,
- **case-study and workflow depth** behind the front page,
- and **carefully selected featured tools** rather than full inventory up front.

A useful framing is:
- the homepage is the magazine cover,
- the deeper pages are the archive and research library.

## Website Structure Direction
A strong website structure for ArchiGen likely includes:

### 1. Homepage
Purpose:
- establish taste,
- establish authority,
- surface strongest imagery,
- point users toward high-value content,
- and frame the platform correctly.

### 2. Learn / Journal / Publication Hub
Purpose:
- articles,
- guides,
- comparisons,
- commentary,
- workflow explainers,

This needs to feel like a real publication, not a forgotten blog tab.

### 3. Tools Directory
Purpose:

- tool profiles,
- categories,
- reviews,
- use cases,
- pricing,

This should support the site, not define the entire brand impression.

### 4. Case Studies / Outputs
Purpose:
- show real results,
- show what the tools can produce,
- build trust,
- make the platform feel grounded.

### 5. About / Positioning Page
Purpose:
- explain the point of view,
- explain why this platform exists,
- connect the site to real architectural workflow.

## Hero Direction
The hero must do heavy lifting.

It should:
- use real architectural renders or outputs,

- one clear supporting statement,

## Recent Website Lessons Already Learned
Several important lessons have already come out of recent homepage work:

1. **Directory energy is the biggest risk**
   If the homepage looks like a directory first, the brand weakens immediately.

2. **Real imagery is necessary**
   Abstract blocks and fake surfaces were not enough. Real render assets improved the direction.

3. **The lower-page hierarchy matters too**
   Even if the hero improves, the rest of the page can still drag the site back toward a generic directory feel.

4. **Editorial framing is stronger than feature-list framing**
   The site works better when it feels like a curated publication rather than a software catalog.

5. **Warmth matters**
   Clean is good, but cold is wrong. The site needs warmth and taste, not just polish.

## Current Design Problem
The main unresolved issue is not whether the site can look nice.
It is whether the whole site system expresses the right identity.

Right now the risk is:
- the hero may move in the right direction,
- but the rest of the homepage still signals tool directory more than premium editorial design platform.

So the build has to focus on full-page hierarchy, not just hero polish.

## Recommended Homepage Hierarchy
A likely stronger homepage hierarchy would be:

1. **Hero with real render imagery**
2. **Short brand framing / what ArchiGen is**
3. **Featured outputs or visual proof block**
4. **What to read / what to use first**
5. **Selected workflow or case study section**
6. **Featured tools section, curated not exhaustive**
7. **Publication / latest articles section**
8. **Newsletter or follow CTA**

This hierarchy keeps the front page curated and aspirational while still connecting into the library.

## What Should Move Off the Homepage
These items should probably live deeper in the site rather than dominate the front page:
- full tool inventories,
- oversized filter systems,
- long category dumps,
- exhaustive lists,
- and overly technical metadata that breaks editorial flow.

## Proof Asset Strategy
The site will be much stronger if it uses real proof assets instead of placeholders.

That includes:
- actual architectural renders,
- workflow screenshots,
- before/after examples,
- tool output comparisons,
- Vista-related work where appropriate,
- and any existing local outputs Quincy has produced.

Real outputs beat abstract branding every time here.

## Website Build Priorities
The order of operations should be:

1. **Fix positioning and page roles**
2. **Fix homepage hierarchy**
3. **Inventory proof assets and imagery**
4. **Refine copy and editorial framing**
5. **Refine tool and article templates**
6. **Tighten the visual system and consistency**
7. **Then polish deployment and launch details**

This keeps the team from polishing the wrong structure.

## Design Heuristics to Follow
Useful rules for this site:
- real outputs beat logos,
- curation beats comprehensiveness,
- proof beats adjectives,
- magazine beats warehouse,
- warmth beats cold polish,
- hierarchy beats feature sprawl,
- page role clarity beats trying to do everything at once.

## What Success Looks Like
The ArchiGen website is successful when it feels like:
- the definitive place for architects exploring AI visualization,
- a site with real judgment and taste,
- a platform that understands both tools and outputs,
- and a premium editorial product rather than a scraped directory.

A visitor should quickly think:
- this is specifically for my field,
- this looks serious,
- these people know what good output looks like,
- and I should spend time here.

## Short Working Summary
ArchiGen AI should be built as a premium, editorial, architecture-first website with real imagery, curated tool selection, workflow depth, and strong visual proof, not as a generic AI tools directory.
